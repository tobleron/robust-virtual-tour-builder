# Public Launch Plan

## Goals

1. Attract hiring managers looking for agentic coding engineers.
2. Attract brokers and teams who want to create their own virtual tours.
3. Turn the repo into a clear public proof of architecture, reliability, and product polish.

## Current Positioning

- The project should read as a serious production codebase, not a demo.
- The README should answer: what this is, how to run it, and what dependencies are required.
- The `_dev-system` docs should explain the analyzer in plain language.
- License and sponsorship text should be policy-oriented and consistent.

## Launch Checklist

### Repo Visibility

- Make the repository public on GitHub.
- Confirm the default branch is the intended public branch.
- Remove or review any obvious temporary or private artifacts before broad promotion.

### Reader-Facing Docs

- Keep `README.md` short at the top and explicit about setup.
- Keep `docs/project/dev_system.md` current with the analyzer settings.
- Keep `_dev-system/README.md` and `_dev-system/ARCHITECTURE.md` readable for external visitors.
- Keep license and sponsorship wording consistent across the repo.

### First-Impression Improvements

- Add a short architecture diagram or quick start in the README.
- Make dependency setup obvious.
- Keep the first screen focused on how to run and trust the project.
- Reduce jargon in public-facing docs.

## Audience Plans

### Hiring / Agentic Coding

- Emphasize ReScript, Rust, analyzer governance, and architectural discipline.
- Highlight the `_dev-system` and task-driven refactor workflow.
- Show that the repo is maintained with build verification and disciplined review.

### Brokers / Tour Users

- Show that the builder is usable locally with `npm run dev`.
- Explain that `./scripts/setup.sh` handles dependency resolution.
- Make the export and viewer workflow easy to understand.
- Keep the broker-facing value proposition simple: create, manage, and export virtual tours.

## Outreach Channels

- GitHub profile and pinned repo.
- LinkedIn post with the architecture story.
- Short post on X/Twitter with screenshots and the quick-start path.
- Real estate and photography communities.
- Agentic coding / AI engineering communities.

## Messaging Rules

- Lead with the product outcome, not the implementation detail.
- Use the analyzer and workflow only as proof of engineering quality.
- Avoid over-selling licensing.
- Keep the tone professional and factual.

## Success Metrics

- GitHub stars
- Forks
- Clones
- Broker inquiries
- Setup completion rate
- Time-to-first-successful-export
- Hiring outreach responses

## Next Actions

1. Finish the README and analyzer docs cleanup.
2. Publish the repository.
3. Create a short launch post and pin the repo.
4. Track traffic and feedback for the first two weeks.
