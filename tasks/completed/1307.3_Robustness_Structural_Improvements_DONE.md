# [1307.3] Robustness Hardening: Tier 3 Structural Improvements

## Context
Part of the broader Robustness Hardening initiative (Task 1307). These are medium-effort structural fixes to improve state management, caching, and concurrency safety.

## Objectives

### Fix 3.1: CircuitBreaker Non-Atomic State Transitions
- **File:** `src/utils/CircuitBreaker.res`
- **Issue:** `canExecute()` mutates `t.state` and `t.probing` as separate assignments. `recordSuccess()` modifies 4 fields independently. In JS this is safe within a synchronous block, but if any of these functions are called from inside a Promise `.then()` while another is mid-execution (possible via reentrant event handling), fields can be inconsistent.
- **Fix:** Replace mutable record fields with an immutable ADT state machine:
  ```rescript
  type cbState = Closed({failures: int}) | Open({since: float}) | HalfOpen({probing: bool, successes: int})
  let state = ref(Closed({failures: 0}))
  ```
  Each function does a single `state := newState` atomic assignment.

### Fix 3.2: Service Worker Stale Cache Strategy
- **File:** `src/ServiceWorkerMain.res`
- **Issue:** Cache-first strategy with NO revalidation. Cached assets are served indefinitely even after app updates. After a deploy, users get stale JS/CSS until the new SW activates and clears old caches. If SW install fails, old cache persists permanently.
- **Fix:** Implement stale-while-revalidate: return cached response immediately but also fetch fresh version and update cache. On next load, user gets latest assets.

### Fix 3.3: Backend Geocoding Cache Potential Deadlock
- **File:** `backend/src/services/geocoding/cache.rs`
- **Issue:** `evict_lru_entry()` holds `GEOCODE_CACHE` write lock while acquiring `CACHE_STATS` write lock. If another thread holds stats lock and waits for cache lock: deadlock.
- **Fix:** Either combine into one `RwLock<(HashMap, CacheStats)>` or release cache lock before acquiring stats lock (copy the key to evict, release, re-acquire).
