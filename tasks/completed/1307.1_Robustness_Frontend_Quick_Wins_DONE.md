# [1307.1] Robustness Hardening: Tier 1 Frontend Quick Wins

## Context
Part of the broader Robustness Hardening initiative (Task 1307). These are high-impact, low-effort fixes for the frontend to eliminate race conditions, data loss risks, and memory leaks.

## Objectives

### Fix 1.1: AsyncQueue Result Ordering Bug
- **File:** `src/utils/AsyncQueue.res`
- **Issue:** `Array.push(results, res)` adds results in COMPLETION order, not submission order. With `maxConcurrency > 1`, results are shuffled.
- **Impact:** Image metadata can be paired with wrong images during upload processing.
- **Fix:** Pre-allocate results array with `Belt.Array.make(total, None)`, write by index `results[i] = Some(res)`, then `Belt.Array.keepMap(results, x => x)` at resolution.

### Fix 1.2: OperationJournal Unwaited Saves
- **File:** `src/utils/OperationJournal.res` (completeOperation, failOperation)
- **Issue:** `saveCurrent()` returns a Promise but is called fire-and-forget. If app crashes before Promise resolves, IndexedDB has stale journal. Recovery system then can't find interrupted operations.
- **Impact:** Recovery system fails to detect incomplete uploads after crash.
- **Fix:** Chain `saveCurrent()` into the returned Promise so callers can await it. Callers already `.then()` on the result, so no API change needed.

### Fix 1.3: PersistenceLayer Data Loss on App Close
- **File:** `src/utils/PersistenceLayer.res`
- **Issue:** 2-second debounce means up to 2 seconds of user work is lost if the app/tab is closed. No `beforeunload` handler flushes pending saves.
- **Impact:** User loses recent edits (scene additions, hotspot moves, etc.) on accidental close.
- **Fix:** Add `window.addEventListener("beforeunload", () => { switch lastSaveTimeout.contents { | Some(_) => performSave(GlobalStateBridge.getState()) | None => () } })` in `initSubscriber()`.

### Fix 1.4: GlobalStateBridge subscribe() In-Place Mutation
- **File:** `src/core/GlobalStateBridge.res`
- **Issue:** `Array.push(listeners.contents, cb)` mutates the array in-place, while unsubscribe creates a new array via `Array.filter`. Inconsistent pattern. If `subscribe()` is called during a `setState()` forEach iteration, the push modifies the array being iterated. While JS forEach won't visit the new element, this is a fragile assumption.
- **Fix:** Change to `listeners := Belt.Array.concat(listeners.contents, [cb])` (matches EventBus pattern).

### Fix 1.5: PersistenceLayer Subscriber Leak
- **File:** `src/utils/PersistenceLayer.res`
- **Issue:** `let _ = GlobalStateBridge.subscribe(onStateChange)` discards the unsubscribe function. During HMR (dev) or if `initSubscriber` is called multiple times, duplicate listeners accumulate, causing N redundant IndexedDB writes per state change.
- **Fix:** Store unsubscribe ref: `let unsub = ref(None)` at module level. In `initSubscriber`: `unsub.contents->Option.forEach(f => f()); unsub := Some(GlobalStateBridge.subscribe(onStateChange))`.

## Verification
- Run `npm run test:frontend` to ensure no regressions.
- Verify that `AsyncQueue` preserves order.
- Verify `PersistenceLayer` saves on window close (manual test).
