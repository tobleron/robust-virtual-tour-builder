# Task 1517: Motion Manifest Contract for Deterministic Cinematic Teaser

## Assignee
Jules (AI Agent)

## Capacity Class
B

## Objective
Create one canonical motion manifest (`motion-spec-v1`) that represents simulation-equivalent teaser movement deterministically, and is consumable by both frontend WebM rendering now and backend MP4 rendering later.

## Depends On
None

## Boundary (Allowed Areas)
- `src/systems/Simulation/`
- `src/systems/Teaser*.res`
- `src/systems/ServerTeaser.res`
- `src/core/JsonParsers*.res`
- `src/utils/Constants.res` (teaser constants only)
- `backend/src/api/media/video.rs`
- `backend/src/api/media/video_logic_support.rs`

## Owned Interfaces
- Teaser motion manifest payload contract between frontend and backend
- Motion-profile parsing and validation utilities
- Teaser request schema fields related to deterministic motion

## No-Touch Zones
- `src/core/Reducer.res`
- `src/core/State.res`
- `src/components/` (except minimal type plumbing if strictly required)
- Export packaging flow (`src/systems/Exporter/`, `backend/src/services/project/package.rs`)

## Problem
Realtime capture paths drift from simulation timing and are hardware-sensitive. A deterministic renderer needs a stable, explicit motion contract independent of runtime frame timing.

## Scope
1. Define `motion-spec-v1` JSON schema with explicit timeline semantics.
2. Include deterministic fields:
   - `fps`
   - `canvasWidth`, `canvasHeight`
   - `includeIntroPan=false`
   - ordered `shots[]` with per-shot:
     - `sceneId`
     - `arrivalPose` (centered on waypoint-start)
     - `animationSegments[]` (yaw/pitch/hfov start/end, easing, durationMs)
     - `transitionOut` (`crossfade`, durationMs)
3. Implement strict decoder/validator on frontend and backend (fail-fast with actionable errors).
4. Add migration-safe versioning (`version: "motion-spec-v1"`), with a clear error for unsupported versions.
5. Add unit tests for valid/invalid payloads and version mismatch.

## Out of Scope
1. Actual deterministic frame rendering.
2. WebM encoding pipeline.
3. MP4 backend rendering implementation.

## Sequential Plan
1. Add schema/types and decoders.
2. Build manifest generation from existing simulation/teaser math.
3. Wire manifest into teaser request payload path.
4. Add backend parser + validation and structured error response.
5. Add tests and docs note.

## Acceptance Criteria
- [ ] `motion-spec-v1` is generated by frontend teaser flow with deterministic shot/segment data.
- [ ] Backend can parse and validate the manifest without runtime pan/scene guesswork.
- [ ] Unsupported/invalid manifest versions fail with explicit, user-safe error messages.
- [ ] Unit tests cover success + failure parsing paths.
- [ ] `npm run res:build` passes.
- [ ] `cd backend && cargo check` passes.
