# 1880 Compact Mobile Landscape Export UI Density and Safe-Area Scaling

## Objective
Improve the exported tour runtime so mobile landscape layouts with short viewport height use a more compact UI density. The goal is to prevent oversized controls and overlays on phones in landscape while preserving the existing touch-friendly behavior.

This task applies to exported tours generated by the repo, not the local builder UI.

## Problem Statement
The current export runtime already distinguishes between:
- `portrait-adaptive`
- `landscape-touch`
- `classic`

However, the landscape-touch UI still feels too large on mobile phones in landscape orientation because the current sizing logic is driven too heavily by overall stage area. A phone in landscape can be wide enough to avoid portrait mode but still have a very short usable height.

That causes several exported-tour UI elements to feel oversized or too dominant:
- floor buttons
- touch mode selector orbs
- persistent labels / looking-mode indicator block
- marketing banner spacing
- watermark/logo offsets and presence
- corner rails and docked controls

## Required Outcome
Add a compact landscape-touch density mode for short-height mobile layouts so exported tours remain easy to operate without visually crowding the viewport.

## Required Changes

### 1. Add Short-Height Landscape Detection
In the export runtime viewport logic, detect when the layout is:
- landscape
- touch-primary
- short-height enough to require compact UI density

This should be based primarily on:
- effective stage height
- short edge / aspect ratio

It should not rely only on total stage area.

### 2. Compact Landscape Density Scaling
When the layout is a short-height mobile landscape case, scale down the exported-tour chrome together as one system.

The compact mode should affect at minimum:
- floor navigation buttons
- touch mode selector orbs
- orb icon size
- orb label font sizes
- orb gaps / docking offsets
- room label / persistent label sizing
- looking-mode indicator sizing and padding
- marketing banner vertical footprint and spacing
- watermark/logo corner offsets and, if needed, size ceiling

### 3. Keep Touch Controls Usable
The compacting must not make the UI too small to tap reliably.

The implementation should:
- reduce visual dominance
- preserve touch usability
- keep controls comfortably tappable on mobile

### 4. Fix Layout Classification Weakness
Review and correct the export viewport classification rule in the runtime so narrow landscape phones are not incorrectly forced into portrait behavior just because the width is under an arbitrary threshold.

The compact landscape-touch mode should be a first-class handled case, not an accidental fallback.

### 5. Safe-Area Awareness
Add safe-area-aware spacing where appropriate so controls and overlays do not sit too close to notches, browser bars, or screen edges.

This should apply especially to:
- bottom controls
- bottom marketing banner
- corner watermark/logo
- left-side floor navigation rail

### 6. Preserve Existing Larger Touch Layouts
Do not make tablets or larger landscape touch devices unnecessarily tiny.

The new compact mode should target:
- phones in landscape
- other very short-height touch layouts

while preserving the current more spacious touch shell for larger layouts.

## Suggested Technical Direction

### Viewport Logic
Extend the export runtime so it can distinguish:
- regular `landscape-touch`
- compact `landscape-touch` for short-height mobile layouts

This can be done by:
- adding a body class for compact landscape touch density
- or adding a compact UI state that drives CSS variables

### Density Variables
Prefer driving the compact mode through CSS variables and shared sizing tokens rather than hardcoding separate one-off rules for each element.

Suggested variable families:
- touch orb size
- floor button size
- font size scale
- horizontal/vertical offsets
- banner height/padding
- logo offset and size cap

### Scaling Inputs
Use inputs that better reflect mobile landscape pressure, such as:
- stage height
- short edge
- aspect ratio

Do not depend only on:
- total stage area

## Acceptance Criteria
- [ ] On a phone in landscape, floor buttons are visibly more compact than the current implementation.
- [ ] Touch selector orbs scale down appropriately in mobile landscape.
- [ ] The logo/watermark placement feels balanced and not oversized in mobile landscape.
- [ ] The marketing banner footprint is tighter in mobile landscape without becoming unreadable.
- [ ] Looking-mode / label overlays do not dominate the scene on landscape phones.
- [ ] Larger tablets in landscape still retain an appropriately touch-friendly layout.
- [ ] Narrow landscape phones are no longer misclassified into the wrong runtime shell because of a width-only rule.
- [ ] Safe-area-sensitive edges have enough padding in mobile landscape.
